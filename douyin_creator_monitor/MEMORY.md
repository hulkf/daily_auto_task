# Project Memory

本文件记录 `douyin_creator_monitor` 项目在多次 Codex 重启、长时间补录和权限审批中的关键经验。后续接手本项目的 Codex 在执行联网、外部目录写入或批量补录前，必须先阅读本文件。

## 2026-07-19：Codex 提权/审批反复失败记录

### 结论

项目脚本是用户本地编写和维护的脚本，但当脚本由 Codex 启动，并且需要联网、调用外部服务、启动本机工具或写入工作区之外的目录时，Codex 命令执行器仍会进行沙箱权限检查。

此前多次失败不是抖音流水线本身报错，而是命令尚未启动时，Codex 的自动审批服务失败。已见到的明确错误包括：

```text
Automatic approval review failed: We're currently experiencing high demand
```

出现这类错误时，Python 任务没有真正开始，不应把它误判成抓取、ASR、IMA、夸克或 Obsidian 卡住。

### 已经发生过的错误做法

1. 把采集、ASR、IMA、夸克、Obsidian 等阶段拆成不同外部命令逐一启动，导致每种命令都可能触发新的提权审批。
2. 用户已经批准固定入口后，Codex 重启恢复任务时仍再次传入 `require_escalated`、新的审批说明和 `prefix_rule`，从而把命令重新送入自动审批服务。
3. 自动审批服务返回 `high demand` 后，重复发起相同提权请求，浪费时间但没有推进流水线。
4. 为了检查代码而发起不必要的新 CodeGraph 提权查询；当 CodeGraph 已有结果或可以在项目工作区内安全读取源码时，不应让代码检查阻塞主任务。

### 后续必须遵守的执行规则

#### 1. 三位博主历史补录只使用固定入口

```powershell
python douyin_creator_monitor\scripts\run_three_creator_backfill.py
```

该入口负责知了、糯米爸、阿笠的断点续跑，并在一个父进程中调用抓取、飞书基础同步、ASR、文案纠正、IMA、夸克、Obsidian、飞书文案/状态回写和达人目录映射子流程；飞书阶段不得跳过。

#### 2. 批准前缀不等于命令已经自动脱离沙箱

当前 Codex 权限上下文中虽然已经列出以下批准前缀：

```text
["python", "douyin_creator_monitor\\scripts\\run_three_creator_backfill.py"]
```

但 2026-07-19 21:03 的实际测试证明：不传 `require_escalated` 原样运行该命令时，进程仍在沙箱内启动，联网被 `[WinError 10013]` 拒绝，Obsidian 外部目录写入被 `[Errno 13] Permission denied` 拒绝。

因此不能再假设“已有批准前缀 = 默认执行会自动获得网络和外部目录权限”。批准前缀只是审批匹配规则；在当前后端中，真正脱离沙箱仍可能需要 `require_escalated`，而该请求又可能被自动审批服务的 `high demand` 故障阻断。

#### 3. 不得改变固定命令的外层形式

不要在固定入口外增加 PowerShell 包装、环境变量赋值、管道、重定向、`cmd /c`、额外 Python 包装脚本或其他间接执行方式。命令字符串改变后可能无法命中既有批准规则，并再次触发审批。

所需环境变量和子进程参数应写在 `run_three_creator_backfill.py` 内部，由固定入口统一设置。

#### 4. 不得拆分申请各阶段权限

不要分别为以下操作申请权限：

- MediaCrawler/抖音联网抓取；
- 火山 ASR；
- IMA 上传；
- 夸克上传；
- Obsidian 外部目录写入；
- 各个博主或单条作品的单独流水线命令。

这些操作必须由固定入口一次性启动并继承同一运行权限。

#### 5. 审批服务异常后的处理

如果固定入口在命令启动前再次出现自动审批服务错误：

- 立即停止重复提权，不尝试包装命令或绕过审批；
- 明确告诉用户“命令未启动，失败发生在 Codex 审批层”；
- 保留本地断点和日志；
- 只有在用户了解本次具体失败并明确要求继续后，才允许再次尝试；
- 再次尝试时仍优先使用已经批准的原始固定命令，不创建新权限规则。

#### 6. 定时任务不应依赖 Codex 审批

正式的每日增量任务应由 Windows 任务计划程序直接运行固定脚本，而不是每天由 Codex 发起。Windows 任务计划程序直接调用本地脚本时，不经过 Codex 自动审批服务。

定时入口应保持非交互、可断点恢复、凭据从本地私有配置读取，并保证登录态和密钥不进入 Git。

### 当前固定入口承担的恢复责任

`run_three_creator_backfill.py` 应长期保持以下行为：

- 发现 `runtime/pipeline/collection/<creator>.json` 中已有 `full_history_collected=true` 时跳过重复全量翻页；
- 只选择缺少最终文案或缺少已启用备份状态的作品；
- 失败作品最多重试有限次数，不能挡住更老的待处理作品；
- 延长火山 ASR 查询等待时间，减少任务实际成功但轮询过早超时；
- 飞书基础同步、最终文案、三处备份状态与达人目录映射必须回写；
- 继续执行 IMA、夸克和 Obsidian；
- 父进程输出必须兼容 Windows GBK 控制台；
- 全程保留日志和断点，可在重启后继续。

## 安全边界

本文件只能记录命令形式、失败原因、流程规则和非敏感状态。不得写入 API Key、Token、Cookie、Session、密码、二维码登录态、个人目录中的凭据内容或其他秘密。

1. 不得再用完整三位博主入口测试“是否有权限”，否则一次权限错误会扩散为大量无效失败日志。

3. 在权限探针完成前，不要再次从 Codex 沙箱直接启动完整历史补录。
4. 如果 Codex 自动审批服务持续不可用，应让用户在本机终端或 Windows 任务计划程序中直接运行固定入口；这种运行不经过 Codex 沙箱审批。

### 本轮修复

- 飞书批量创建不能只按 200 条分批。Windows `CreateProcess` 会因巨大 `--json` 参数触发 `[WinError 206]`；现在同时限制每批最多 200 条、JSON 最多 20,000 字符，并保持返回 `record_id` 与作品顺序一致。
- 抖音 SecUID 可能超过飞书 CLI 关键词搜索的 50 字符限制；长 SecUID 必须改用 `record-list` 分页后精确匹配。
- 历史达人目录映射 marker 若记录 `feishu_writeback=disabled`，不得视为已完成缓存。
- `--feishu-only` 的进程状态只由飞书相关操作决定；本地 IMA 失败仍如实回写，但不再导致飞书补写任务误报失败。

## 2026-07-19 性能优化决策

主流水线采用“双流水线 + 达人级批量收口”：目录映射与 ASR 并行，ASR 完成一条即按完成顺序进入单消费者执行 IMA/本地知识库等可即时交付环节；夸克在达人级用 manifest 批量上传，随后飞书最终结果按达人批量写回。永久错误熔断仅在当前达人、当前交付目标内生效，不跨达人继承；超时、429、5xx、DNS 等临时错误不熔断。最终结果仍按原作品顺序汇总，确保报告稳定和断点状态可追踪。
