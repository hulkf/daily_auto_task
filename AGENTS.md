# Daily Auto Task Workspace Instructions

本目录是一个总任务仓库，用来沉淀和运行日常工作自动化任务。后续在 Codex 中新建任务、维护脚本、接入定时调度时，应优先读取并遵守本文件。

## 1. 任务读取与意图识别

- 当前仓库的核心意图是：把用户每天重复执行的工作流程整理成可复用、可运行、可调度的自动化任务。
- 新任务开始时，先理解用户要自动化的具体工作，包括输入来源、处理步骤、输出结果、运行频率、是否需要人工确认。
- 如果用户没有明确指定子项目目录，应先判断该任务属于已有子项目，还是应该创建新的独立子项目。
- 如果任务目标、数据来源、账号权限、执行频率或输出位置不清楚，应先补充确认关键问题，再进行实现。
- 优先把一次性的操作沉淀为可复用脚本、配置或文档，方便后续改造成定时任务。

## 2. 文件夹管理

- 每一个不同项目或不同业务目标，都应放在独立文件夹中，避免代码、配置、运行产物相互干扰。
- 推荐的新项目目录放在 `projects/` 下，例如：

```text
projects/
  project_name/
    AGENTS.md
    README.md
    src/
    data/
    output/
    logs/
    config/
```

- `src/` 用于放项目脚本或程序代码。
- `data/` 用于放输入数据、原始文件或待处理素材。
- `output/` 用于放生成结果。
- `logs/` 用于放运行日志和排错记录。
- `config/` 用于放配置模板、示例配置和非敏感配置。
- 每个子项目可以有自己的 `AGENTS.md`，用于记录该子项目的特殊规则。子项目说明优先级高于本文件。
- 不要把不同项目的临时文件、账号状态、缓存或输出混在同一个目录里。
- 公共工具可以放在 `tools/` 或 `shared/`，但只有在多个项目确实复用时才抽出来。

## 3. 总任务仓库与自动化调度

- 本仓库是日常自动化任务的总入口，目标是让不同任务可以独立维护、独立运行、独立排错，并在未来接入定时调度。
- 新增任务时，应尽量设计为命令行可运行的形式，方便以后被 Windows 任务计划程序、cron、CI、调度平台或其他自动化入口调用。
- 每个可调度任务应尽量具备：
  - 明确的启动命令。
  - 明确的输入和输出目录。
  - 可配置的运行参数。
  - 日志输出。
  - 失败时可读的错误信息。
  - 不依赖手动打开编辑器或临时交互。
- 涉及登录态、二维码、token、cookie、密钥等敏感或易变状态时，应放在对应子项目的独立配置或运行目录中，不要混入其他项目。
- 自动化任务之间默认互不依赖。只有用户明确要求串联时，才建立跨项目调用关系。
- 修改已有项目时，只改该项目目录内的文件。若必须修改公共工具或总配置，应说明影响范围。

## 4. 后续 Codex 工作规则

- 开始任何实现前，先查看当前目录结构和相关子项目说明。
- 优先复用已有脚本、配置和目录约定，不随意引入新的组织方式。
- 对运行路径、生成文件、调度入口等用户实际会使用的内容，要给出明确位置和命令。
- 如果创建了新子项目，应同时补充该子项目的 `README.md` 或 `AGENTS.md`，说明用途、入口命令、输入输出位置。
- 对于可能长期运行或定时运行的任务，优先考虑稳定性、日志、幂等性和失败恢复。

## 5. Git 推送与敏感信息保护

- 本项目后续会推送到 Git，推送前必须严格检查代码和待提交文件中是否包含个人密钥、账号凭据或登录状态。
- 每个任务或子项目发生有效代码、配置模板、文档或调度入口改动后，应及时提交并推送到 GitHub，避免本地改动长期滞留。
- 及时推送不代表跳过检查。任何提交和推送都必须先完成敏感信息检查，再执行 `git add`、`git commit`、`git push`。
- 严禁提交真实的 API Key、Access Token、Refresh Token、Secret Key、Cookie、Session、二维码登录态、私钥、密码、手机号验证码、浏览器用户数据目录、`.env` 实际配置文件等敏感内容。
- 只允许提交核心源代码、非敏感配置模板、示例数据、说明文档和必要的任务入口脚本。
- 如果代码需要密钥或账号配置，应提供 `.env.example`、`config.example.json` 或 README 说明，用占位符表示，例如 `YOUR_API_KEY_HERE`。
- 如果某个任务必须保存本地登录态或运行缓存，应放在子项目自己的 `runtime/`、`secrets/`、`local/` 或类似目录，并确保这些目录不进入 Git。
- 推送前必须执行敏感信息检查，至少包括：
  - 查看 `git status --short --untracked-files=all`，确认待提交文件范围。
  - 搜索常见敏感关键词，例如 `token`、`secret`、`api_key`、`apikey`、`access_key`、`cookie`、`session`、`password`、`authorization`、`bearer`、`private_key`。
  - 检查 `.env`、配置文件、日志文件、运行产物和二维码图片是否被误加入版本控制。
- 如果发现敏感信息，先移出 Git 提交范围或替换为占位符，再提交和推送。
- 如果敏感信息已经进入 Git 历史，应停止推送，先询问用户是否需要清理 Git 历史，并提示用户立即轮换相关密钥。
- Codex 在执行 `git add`、`git commit`、`git push` 前，应主动进行上述检查并向用户说明检查结果。
- Codex 完成一个任务的文件改动后，应优先询问或执行提交推送流程；如果当前目录还没有初始化 Git 或没有 GitHub 远程仓库，应明确告知用户需要先完成仓库初始化和远程地址配置。
